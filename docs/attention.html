Scaled Dot-Product Attention (naive)

TL;DR. Three kernels back to back: build the [M × N] scores matrix in DRAM, run row-wise softmax on it in place, then multiply by V to produce the output. The middle matrix is the bandwidth villain — it gets written to DRAM, read for softmax, read again for the output multiply. Flash attention's whole reason for existing is to avoid this round trip.

Source: /workspaces/leetgpu-master/attention.cu.

DRAM L2 L1 / shared register warp

1. The math, in 3 lines

scores = (Q · K^T) / sqrt(d)            // shape [M, N]
P      = softmax_along_rows(scores)     // shape [M, N]
O      = P · V                          // shape [M, d]

Shapes: Q is [M × d], K and V are [N × d]. For the test harness M = N = 512, d = 64. So we build a 512×512 = 262,144-entry scores matrix in the middle. Remember that number.

2. Why three kernels (again)

Same constraint as softmax: blocks on different SMs can't synchronize. We need a global barrier between "scores done" and "softmax starts", and another between "softmax done" and "output multiply starts". The cheapest barrier is a kernel boundary.

Pipeline. The [M×N] scores buffer is allocated once and reused (attention.cu:166).

3. Kernel 1 — scores

One thread = one element of the [M × N] scores matrix. The thread loads Q[i] and K[j] (both length d), does the dot product in a single thread, and writes the result. 

__global__ void attentionScoresKernel(const float* __restrict__ Q,                                       const float* __restrict__ K,                                       float* __restrict__ scores,                                       int M, int N, int d, float scale) {    int j = blockIdx.x * blockDim.x + threadIdx.x;     // column = key index    int i = blockIdx.y * blockDim.y + threadIdx.y;     // row    = query index    if (i >= M || j >= N) return;    float dot = 0.0f;    const float* q = Q + i * d;    const float* k = K + j * d;    for (int e = 0; e < d; e++) dot += q[e] * k[e];    scores[i * N + j] = dot * scale;}

Block size 16×16 = 256 threads. Within a warp, threadIdx.x varies (so j varies, i is fixed). That means all 32 threads in a warp share the same Q row but read different K rows.

Memory access cost (one warp)

  • Q row: 32 threads all read the same i'th row. The hardware broadcasts — one load fills all 32 threads via L1/L2.
  • K rows: 32 threads read 32 different rows of K. For each e, that's a strided read across DRAM — uncoalesced. With d inner iterations, this kernel is bandwidth-bound on the K reads.
  • Compute: d multiply-adds per element × M·N elements = 2·M·N·d FLOP. For M=N=512, d=64 that's ~33 MFLOP — trivial on a 5 TFLOPS card.
A real attention kernel tiles Q and K into shared memory and does block-matrix products (essentially a GEMM). The naive version above leaves K's reads uncoalesced — easy to read, slow to run.

4. Kernel 2 — row-wise softmax

One CTA per row of scores (grid size = M). Inside the block, 256 threads cooperate to do the same three-pass softmax we saw on the softmax page: row-max reduction, exp + sum reduction, normalize. Three __syncthreads()-separated phases. 

__global__ void softmaxRowsKernel(float* __restrict__ scores, int M, int N) {    int row = blockIdx.x;                       // one CTA per row    float* s = scores + row * N;    int t = threadIdx.x;    __shared__ float buf[256];    // Phase 1: row max via tree reduction    float local_max = -FLT_MAX;    for (int j = t; j < N; j += blockDim.x) local_max = fmaxf(local_max, s[j]);    // … (tree reduction in shared mem, then row_max = buf[0])    // Phase 2: exp + sum (writes exp(s[j]-row_max) back into s, accumulates sum)    // Phase 3: normalize by inv_sum}

Why one CTA per row? Because a row is self-contained — the softmax denominator only depends on values in that row. Different rows have no data dependency, so they can run on different SMs in parallel without needing any cross-block sync.

Each CTA owns one row. M rows = M CTAs = M parallel softmaxes.

5. Kernel 3 — output

Same shape as Kernel 1, but now we walk over j inside the thread instead of e. One thread = one output element O[i][k]: 

__global__ void attentionOutputKernel(const float* __restrict__ P,                                       const float* __restrict__ V,                                       float* __restrict__ O,                                       int M, int N, int d) {    int k = blockIdx.x * blockDim.x + threadIdx.x;     // output-dim index    int i = blockIdx.y * blockDim.y + threadIdx.y;     // query index    if (i >= M || k >= d) return;    float acc = 0.0f;    const float* p = P + i * N;    for (int j = 0; j < N; j++) acc += p[j] * V[j * d + k];    O[i * d + k] = acc;}

Two big DRAM traversals here:

  • P: the whole row of softmaxed probabilities — same one for every k within a warp, so it broadcasts.
  • V: we read every row of V × N entries. Over the whole kernel, V is read M times (once per query row).

6. The bandwidth disaster, drawn

Look at every byte that crosses the DRAM bus during one attention call. For M = N = 512, d = 64, scores is 1 MB (262,144 × 4 B).

Three kernels, three trips through DRAM for the scores buffer.
step 1 / 5
The fix: don't materialize the [M × N] matrix. Fuse all three kernels into one. Each CTA owns one query row, walks across all N keys once, and keeps three running scalars in registers instead of writing the scores buffer:
  • m — running max of scores seen so far
  • l — running denominator Σ exp(score − m)
  • acc — running output Σ softmax_weight × V[j]
When a new key has a bigger score, both l and acc get rescaled by exp(m_old − m_new) in a single multiply — provably equivalent to the "real" softmax. After N keys, write O[i] = acc / l.

The trade: K and V get re-read once per query (slightly more DRAM traffic on those tensors) in exchange for eliminating all 4 MB of scores-buffer round-trips. Walk through it on the flash attention page →

Two further optimizations production versions stack on top of flash attention: tile multiple queries per CTA (so K and V are re-used across the tile, eliminating the "re-read per query" cost) and tensor cores for the dot product (not on this card — the 1660 Super has none — but on RTX 30xx+ that's where the headline FLOPS speedup comes from).

The [M × N] scores matrix is the bandwidth villain. Even though softmax re-reads it from L2 instead of DRAM (it fits when small), the write from kernel 1 and the read by kernel 3 both go through DRAM. For M=N=4096 (a realistic attention head), scores is 64 MB — blows L2 (1.5 MB) completely, every kernel pays the full DRAM round trip.

8. Why this design exists at all

It's the textbook decomposition. Each piece maps cleanly to a kernel, each kernel is easy to verify, debugging is straightforward. The cost is the materialized middle matrix.

For small attention heads (this test case fits comfortably in L2) it's competitive. For real transformer workloads (seq_len 2048+) it falls off a cliff because scores stops fitting in any cache.

9. Pitfalls

  • Forgetting the scale factor — without / sqrt(d) the softmax saturates to a one-hot for large d.
  • Numerical instability without max-trick — kernel 2 does it correctly (line 104, 117).
  • K accessed row-by-row in kernel 1 with stride d — if d is small (e.g. 64) and aligned, the inner loop may coalesce within a warp; if d isn't a multiple of 32, the access pattern degrades.
  • Memory blowup at long sequence: scores grows as N²; at N=8k it's 256 MB — won't even fit in the 6 GB GDDR6, much less L2.
