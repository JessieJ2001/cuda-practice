Flash Attention

TL;DR. Same math as naive attention, but the entire pipeline runs in one fused kernel that never materializes the [M × N] scores matrix. Each CTA owns one query row and walks across all N keys once, updating three tiny running scalars (m, l, acc) after each key. The price: K and V are re-read N times. The win: sequence lengths that would otherwise blow out DRAM (and L2) just keep working.

Source: /workspaces/leetgpu-master/flash_attention.cu.

DRAM L2 L1 / shared register warp

1. The core insight

Naive attention has to build the full scores matrix because softmax needs the whole row to compute its denominator. You can't normalize until you've seen every key.

Flash attention's trick is the online softmax: as you visit keys one at a time, maintain

VariableMeaningLives in
mrunning max of scores seen so farregister (per-thread, redundantly)
lrunning sum of exp(score − m)register
accrunning sum of weight × V for this thread's output dimregister

Whenever a new key produces a larger score, the previous accumulators are rescaled by exp(m_old − m_new) — math that's equivalent to having normalized with the right max all along. After N keys, divide acc by l and write to output. Done.

Total extra state: O(d) per CTA (one register slot for each output dim). The naive version's [M × N] matrix becomes nothing.

2. The math, with proof of equivalence

For one query i and keys j = 0..N-1: 

true softmax:    p_j = exp(s_j - row_max) / Σ_k exp(s_k - row_max)
                 O   = Σ_j p_j · V[j]

flash form:      after key j, given (m, l, acc) reflecting keys 0..j-1:
                 m_new   = max(m, s_j)
                 corr    = exp(m - m_new)        // ≤ 1, rescales old state
                 p_j_raw = exp(s_j - m_new)
                 l       = l · corr + p_j_raw    // new running sum
                 acc     = acc · corr + p_j_raw · V[j]
                 m       = m_new

after all N:     O = acc / l                     // ≡ true softmax above

Why it's equivalent: every term in acc and l is consistently expressed relative to the current max m. When m changes, both numerator and denominator get the same corr multiplier, so the final ratio acc / l is unchanged.

3. The whole kernel, annotated

__global__ void flashAttentionKernel(const float* __restrict__ Q,                                      const float* __restrict__ K,                                      const float* __restrict__ V,                                      float* __restrict__ O,                                      int M, int N, int d, float scale) {    const int i = blockIdx.x;                  // one CTA per query row    const int t = threadIdx.x;    const int T = blockDim.x;                  // power-of-2 ≥ d    extern __shared__ float smem[];    float* q_sh = smem;                       // [d] query vector for row i    float* red  = smem + d;                   // [T] reduction scratch    if (t < d) q_sh[t] = Q[i * d + t];        // load Q row once    __syncthreads();    float m   = -FLT_MAX;                      // running max    float l   = 0.0f;                          // running denominator    float acc = 0.0f;                          // running output (for this thread's k = t)    for (int j = 0; j < N; j++) {        // dot product Q[i] · K[j] via a block reduction over the d dim        red[t] = (t < d) ? q_sh[t] * K[j * d + t] : 0.0f;        __syncthreads();        for (int s = T >> 1; s > 0; s >>= 1) {            if (t < s) red[t] += red[t + s];            __syncthreads();        }        const float s_j = red[0] * scale;             // every thread reads the same broadcast        // streaming softmax update — all in registers, all in one cycle        const float m_new = fmaxf(m, s_j);        const float corr  = __expf(m - m_new);        const float p     = __expf(s_j - m_new);        l = l * corr + p;        if (t < d) acc = acc * corr + p * V[j * d + t];        m = m_new;        __syncthreads();                          // before overwriting red[t] next iter    }    if (t < d) O[i * d + t] = acc / l;            // final normalize}

Launch config (flash_attention.cu:155): grid = M (one CTA per query), block = nextPow2(d) (so the dot-product reduction is clean), shared = (d + T) × 4 B (query cache + reduction scratch). For d = 64, that's 64 threads/block and ~0.5 KB shmem/block.

5. Memory comparison (this is the whole point)

Naive attentionFlash attention
[M × N] scores buffer4 MB written, 4 MB read (for M=N=512)0 — never exists
Q readsper element of scores (~N × per Q row)once per CTA (cached in shared)
K readsper element of scores → N × d per Q rowN × d per Q row (re-read each query)
V readsper element of output → M × N × dM × N × d (same trade)
Kernel launches31
Per-CTA extra state—O(d) registers
The trade. Flash attention re-reads K and V N times per query — that's more DRAM traffic on those tensors compared to a well-tiled naive attention. But it eliminates the [M × N] traffic entirely. For long sequences (large N) the scores buffer wouldn't fit in cache anyway, so removing it is a pure win. For tiny problems (this test case) the naive version can actually be faster — the test harness comparison is instructive.

6. Optimizations the source doesn't include

The simplified version in flash_attention.cu processes one query row per CTA (Br = 1 in FlashAttention terms). Production FlashAttention tiles Br queries together, which:

  • amortizes K/V reads across Br rows (huge: K is re-read N times instead of N × M times)
  • uses tensor cores for the dot product (not applicable on TU116, but that's what gets cited in benchmarks)
  • uses warp-level shuffles instead of __syncthreads()-heavy block reductions

The version on the page is the cleanest possible introduction to the online-softmax math; the production version is the same math wrapped in progressively more optimization layers.

7. Pitfalls

  • Forgetting to __syncthreads() at the end of each j-iteration — the next iteration's red[t] = ... would race with threads still reading red[0] for s_j. The source does this at line 134.
  • Using exp() instead of __expf() in the inner loop costs significant cycles. The fast intrinsic is accurate enough (1 ulp) for softmax.
  • Threads t ≥ d are padding for the reduction. They must contribute 0 (line 118 does this with the ternary), but they still participate in the syncs and reductions.
  • Choosing T: next-pow2-of-d, but if d > 1024 the kernel can't launch — block size is capped. Production FlashAttention loops over the d dimension to handle bigger head sizes.
