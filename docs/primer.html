2. The bigger picture — where the GPU sits

Before zooming into the chip, it helps to see the whole machine. A GPU is not the computer — it's an accelerator hanging off a CPU. Data starts life in CPU memory, crosses a PCIe link into the GPU's own DRAM, and only then can the SMs touch it. In a training rig there are usually several GPUs wired together with a much fatter link (NVLink).

CPU ↔ GPU over PCIe; GPU ↔ GPU over NVLink. Inside a GPU: HBM → L2 → SMs.

Three links, three speeds

  • PCIe (CPU ↔ GPU) is the narrowest pipe — tens of GB/s. Moving a tensor across it is expensive, so you copy inputs to the GPU once and keep them there. On datacenter parts PCIe is ~128 GB/s bidirectional; on this card it's less. Either way it's dwarfed by on-GPU bandwidth.
  • NVLink (GPU ↔ GPU) is a wide link (up to 900 GB/s per GPU on NVLink-equipped datacenter cards) used to shard a model that doesn't fit on one GPU. Consumer cards like the 1660 Super don't have it.
  • HBM / GDDR6 (the GPU's own DRAM) is what CUDA calls global memory. It's the biggest, slowest tier on the GPU, and it's where your input tensors and weights live. Every SM can read it; the entire rest of this primer is about not reading it more than you have to.
Trend to keep in mind: across GPU generations, compute gets much faster while memory bandwidth grows only a little. That gap is why so many kernels are memory-bound, and why the optimizations here — coalescing, shared-memory tiling, fusion — keep paying off.

3. The memory hierarchy, to scale

Every byte you touch lives somewhere on a hierarchy: the higher up, the faster — and the smaller. CPUs and GPUs share the same shape (registers on top, DRAM at the bottom) but make opposite bets. A CPU spends its transistors on a deep cache stack (L1/L2/L3) to make a few threads' memory accesses low-latency. A GPU spends them on a huge register file and programmer-managed SRAM so thousands of threads can hide latency instead. The whole game of GPU programming is moving the right bytes up this hierarchy and reusing them before they fall back down.

Same shape, opposite bets: the CPU invests in a deep L1→L2→L3 cache stack; the GPU drops L3, grows the register file, and hands you the SRAM as programmer-managed shared memory.

Read across a row and the contrast is sharp. The GPU's register file is ~100× larger per core-equivalent than a CPU's — it has to be, to hold state for thousands of resident threads. Its DRAM bandwidth is ~5× higher (336 GB/s vs ~60 GB/s) because throughput, not latency, is the point. And it has no L3 at all: where a CPU leans on a big shared last-level cache to keep a handful of threads fast, the GPU expects you to stage data explicitly into shared memory and to keep enough warps in flight that DRAM latency simply disappears behind other work.

The same tier has many names. The GPU's DRAM row alone goes by half a dozen: it's the physical memory (HBM on datacenter cards, GDDR6 here), which CUDA calls global memory / GMEM / device memory. They all point at the same bottom tier. Watch for these aliases across the site and in NVIDIA's docs:
Tier in the diagramOther names you'll seeWhat it physically is
DRAM HBM · GDDR6 · global memory · GMEM · device memory Off-chip DRAM — the GPU's own main memory. Everything starts here.
L1 / Shared SRAM · shared memory · SMEM · scratchpad On-SM SRAM; the programmer-managed part is "shared memory," the automatic part is the L1 cache.
L2 L2 cache · last-level cache The single chip-wide hardware cache in front of DRAM.

The latencies above are approximate — the real numbers vary with contention — but the order of magnitude gap is what matters. Touching DRAM costs roughly 500× more cycles than touching a register.

The GPU hierarchy, tier by tier

Zooming back in on the GPU side: every tier is a trade of the same currency, capacity for bandwidth. The register file is blindingly fast but tiny; DRAM is huge but slow. There is no tier that is both big and fast — if there were, we'd only build that one.

Register and L1/shared bandwidth are aggregate across the SM's ports and vary with access pattern — treat them as "order-of-magnitude faster than the tier below," which is the number that actually drives decisions. Datacenter parts (H100) keep the same ordering but multiply every figure: 3.35 TB/s HBM, 31 TB/s SRAM, 370 TB/s registers.

Two things this hierarchy forces on you

  • Reuse beats re-fetch. A byte pulled from DRAM should do as much work as possible before you drop it. Loading a value into shared memory or a register and using it once is nearly as wasteful as not caching at all — the win comes from touching it many times while it's up the pyramid. This is the entire motivation for tiling.
  • Higher tiers are small, so you stage deliberately. You can't park a whole matrix in 96 KB of shared memory. You bring up a tile, work it, evict it, bring up the next. Choosing tile sizes that fit the SRAM budget — without killing occupancy — is the core craft of a fast kernel.
Shared memory vs L1 — same silicon, different control. On this card the 96 KB per SM is one physical block of SRAM. Part of it acts as an automatic L1 cache (hardware decides what to keep); the rest is shared memory — a scratchpad you address explicitly and that every thread in the CTA can see. The split is configurable. Shared memory is how a CTA's threads cooperate; L1 is a safety net for everything else.
Registers can spill. If a thread needs more live values than its register budget, the compiler "spills" the overflow to local memory — which, despite the name, is just a private slice of slow DRAM. A kernel that looks compute-heavy but runs slow is often quietly spilling; watch your register count.

4. How threads are organized — grid, CTA, warp, thread

When you launch a CUDA kernel with kernel<<<gridDim, blockDim>>>(…), you're saying: "run this function this many times, in parallel." The threads are grouped into a four-level hierarchy:

GridAll threads in one kernel launch.Thousands to millions

How a thread finds its place — the four built-ins

Every thread runs the same kernel code, so the only way it knows which piece of the work is its piece is by reading four built-in variables the hardware fills in for it:

Built-inMeaning
threadIdxIndex of this thread within its CTA.
blockIdxIndex of this thread's CTA within the grid.
blockDimNumber of threads per CTA (the blockDim you launched with).
gridDimNumber of CTAs in the grid (the gridDim you launched with).

The classic one-liner int i = blockIdx.x * blockDim.x + threadIdx.x; turns "which CTA am I in, and where inside it" into a single global index. All of these can be 1-, 2-, or 3-dimensional (.x, .y, .z), which is handy for tiling 2-D matrices.

One thread ≠ one output element. It's tempting to assume every thread computes exactly one result, but you're free to choose any mapping. A 32×32 CTA is 1024 threads; you might have each thread compute several outputs, or a whole warp cooperate on one. The mapping from threads to data is your design decision, not a hardware rule.

Why this matters

  • A CTA never splits across SMs. All its threads live together — they can share fast SRAM and synchronize with __syncthreads().
  • A warp is the real unit of execution. The hardware schedules warps, not individual threads. When you call a kernel, what actually runs is "warp 0 issues an instruction; warp 1 issues an instruction; …" — 32 threads at a time.
  • Threads inside a warp run in lockstep. They all execute the same instruction on the same cycle. If 31 are doing useful work and 1 is idle, the warp still takes one cycle.
  • Pick CTA sizes that are multiples of 32. A CTA of, say, 100 threads still rounds up to 4 warps (128 lanes) with 28 lanes wasted every cycle. In practice people use 128, 256, or 512.

5. The SM (Streaming Multiprocessor)

An SM is one of the 22 "mini-processors" on the chip. Think of it as a tiny multicore CPU with its own private cache.

Inside one SM. The 22 of these on the chip work fully independently.

Each SM picks a ready warp every cycle and issues one instruction to all 32 of its threads. When a warp stalls (waiting on a slow DRAM load, say), the scheduler instantly switches to another warp that is ready. With enough warps in flight per SM, the GPU hides the 500-cycle DRAM latency completely — this is called latency hiding, and it's why GPUs care less about caches than CPUs do.

Occupancy — how many warps you keep on tap

Latency hiding only works if there are other warps ready to run. An SM can hold up to 2048 threads (64 warps) resident at once, and several CTAs share it. But two hard budgets cap how many actually fit:

  • Registers. The SM's ~64K registers are split across every resident thread. A kernel that needs 64 registers/thread can keep all 2048 threads alive; one that needs 128 registers/thread can only keep 1024.
  • Shared memory. The 96 KB (228 KB on an H100) is split across resident CTAs. A CTA that grabs 48 KB means at most two CTAs fit per SM.
Occupancy = how many warps (or CTAs) are concurrently resident on an SM, relative to the hardware max. Higher occupancy gives the scheduler more warps to hide latency with — but it's a means, not a goal. A tiling kernel with tons of register reuse can run fast at low occupancy. Chase latency hiding, not the occupancy number itself.

One more detail worth carrying into the kernel pages: bandwidth climbs at every tier, not just latency. On an H100 the numbers are stark — HBM moves ~3.35 TB/s, SRAM ~31 TB/s, and the register file ~370 TB/s. Same ordering on this card, smaller magnitudes. Keeping a value in registers or shared memory instead of re-fetching it from DRAM isn't a rounding error; it's a 100×-bandwidth decision.

Datacenter aside: Hopper/Ada SMs add a Tensor Core — a dedicated matrix-multiply unit (an H100 hits ~989 TFLOP/s of matmul, vs ~67 TFLOP/s of plain FP32). The 1660 Super has none, so on this card every FLOP goes through the 64 CUDA cores. When you read about "keeping the Tensor Core busy," that's a datacenter concern; here the equivalent goal is keeping the CUDA cores fed.

6. Warps and SIMT — one instruction, 32 threads

SIMT = "Single Instruction, Multiple Threads". The hardware fetches one instruction and broadcasts it to 32 threads. Each thread executes that instruction on its own data, in its own registers. As long as all 32 threads want to do the same thing, everyone wins.

The catch is divergence. If your code has an if that sends some threads down one branch and others down a different branch, the warp has to execute both branches serially, masking off the threads that aren't participating. A heavy if/else inside a hot loop can halve your throughput.

7. Memory coalescing — the single most important optimization

When a warp issues a load, the hardware tries to pack all 32 reads into as few DRAM transactions as possible. The rule of thumb:

If 32 threads in a warp read 32 consecutive 4-byte values starting at an aligned address, that's one 128-byte transaction.
If those 32 threads read 32 values scattered around DRAM, the hardware has to issue up to 32 separate transactions — about 32× the bandwidth cost.

On a 336 GB/s memory bus, that's the difference between a kernel that runs at full speed and one that's 32× slower. The figure below makes it concrete.

8. Shared memory & bank conflicts

Shared memory (the programmer-managed slice of that 96 KB per-SM SRAM) is fast, but it's split into 32 banks, each 4 bytes wide and striped across addresses. Bank k holds words k, k+32, k+64, ….

A warp can read 32 different banks in one cycle. If two threads in the same warp hit the same bank (different addresses), the hardware serializes them. In the worst case, all 32 threads hit the same bank — a 32-way bank conflict — and the access takes 32× longer.

9. Mapping work to hardware — tiling, CTAs, and waves

Take a matrix multiply C[M×N] = A[M×K] · B[K×N]. You don't launch one thread per output — you chop C into rectangular tiles and hand one CTA per tile. Each CTA loads its slice of A and B into shared memory, multiplies, and writes its tile of C.

The number of tiles is your grid size. If C splits into a 3×8 arrangement of tiles, that's gridDim = 24 CTAs. Those 24 CTAs are independent, so the hardware sprays them across SMs:

  • Inter-SM: in parallel. Different CTAs run on different SMs at the same time.
  • Intra-SM: time-sharing. Several CTAs may be resident on one SM (occupancy permitting), and its warp scheduler interleaves them.

Because an SM runs a bounded number of CTAs at once, the grid is processed in waves: the SMs grab a batch of CTAs, finish, then grab the next batch. And here's the trap — if your CTA count spills just past a wave boundary, you pay for a whole extra wave while most SMs sit idle. This is wave quantization (the "tail effect").

10. Arithmetic intensity — compute-bound vs memory-bound

Every kernel is limited by one of two things: how fast the cores can compute, or how fast memory can feed them. Which one? Compare the work done per byte moved:

Arithmetic intensity (AI) = FLOPs performed ÷ bytes moved from DRAM. Compare it to the machine's ridge point = peak FLOP/s ÷ DRAM bandwidth.

On this card that's 5.0 TFLOP/s ÷ 336 GB/s ≈ 15 FLOP/byte. (An H100 sits far higher: 989 TFLOP/s ÷ 3.35 TB/s ≈ 295 FLOP/byte — its enormous Tensor Core throughput makes it much harder to stay compute-bound.)

  • AI < ridge → memory-bound. The cores are starved; you're limited by DRAM bandwidth. Optimize the memory traffic.
  • AI > ridge → compute-bound. Memory keeps up; you're limited by raw FLOP/s. Optimize the arithmetic.
The roofline: a compute ceiling, a memory slope, and the ridge where they meet.

Why tiling and fusion help

Both of these raise arithmetic intensity by reusing bytes instead of re-fetching them: a shared-memory tile loads a chunk of DRAM once and does many FLOPs on it; fusion (as in flash attention) avoids writing a big intermediate to DRAM and reading it back. A naive vector add does ~1 FLOP per 12 bytes — hopelessly memory-bound. A well-tiled matmul reuses each loaded value across a whole tile, pushing AI past the ridge and onto the compute ceiling.

MFU vs MBU. Two utilization scores tell you how close you are to a roof. Model FLOPs Utilization = achieved FLOP/s ÷ peak FLOP/s — how busy the cores are. Model Bandwidth Utilization = achieved GB/s ÷ peak GB/s — how busy the memory bus is. A compute-bound kernel chases high MFU; a memory-bound kernel chases high MBU. You almost never max both.
