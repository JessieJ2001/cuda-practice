Softmax

TL;DR. Softmax forces you to compute two global values across the whole array (max and sum-of-exp) before any output can be written. CTAs can't talk to each other mid-kernel, so the work is split into three kernel launches. Inside each kernel, threads cooperate via a tree reduction in shared memory, then one thread per CTA atomically adds the block result to a global scalar. The numerical-stability trick (subtract the max) keeps exp() from overflowing.

Source: /workspaces/leetgpu-master/softmax.cu.

1. The problem

Given a vector x of length N, softmax produces a probability vector p where: 

p[i] = exp(x[i] - max(x)) / Σ exp(x[j] - max(x))     // numerically stable form

Three things to notice:

  1. You need max(x) before you can start exp.
  2. You need Σ exp(...) before you can divide.
  3. The subtraction by max isn't math — it's defensive: without it, exp(50) overflows to +inf and your output becomes nan.

2. Why three kernels?

Here's the constraint that shapes the entire design: CTAs on different SMs cannot synchronize during a kernel. There's no __syncthreads() across blocks. The only way to wait for all CTAs to finish a phase is to end the kernel — the next kernel launch is the global barrier.

Softmax has two such phases (compute max, compute sum), so we get three kernels:

Three kernels, two global scalars passed between them. From softmax.cu:131–140.

3. The pattern inside one CTA — tree reduction

Both reducing kernels use the same trick: each CTA folds its slice of the array in half, repeatedly, in shared memory. After log₂(blockDim.x) folds, thread 0 holds the block's contribution. That contribution gets atomicAdd'd to a global scalar.

Here's the inner loop from softmaxPartial (softmax.cu:57–68):

data[threadIdx.x] = unnorm;                                // stage in sharedfor (int range = blockDim.x / 2; range >= 1; range /= 2) {    if (threadIdx.x < range)        data[threadIdx.x] = data[threadIdx.x] + data[threadIdx.x + range];    __syncthreads();                                      // wait for all writes}if (threadIdx.x == 0) {    float value = data[0];    atomicAdd(exp_sum, value);                              // publish block result globally}

Each pass halves the active thread count. With blockDim.x = 256, the reduction takes 8 passes (256 → 128 → 64 → … → 1). Step through it:

Two important properties of this pattern:

  • No bank conflicts. At each pass, the active threads access addresses [0..range) and [range..2·range) — both contiguous, both coalesced into shared-memory banks.
  • Warp divergence shrinks each pass. Once range < 32, only thread 0..(range-1) of the warp does work while the rest sit idle. This is the unavoidable cost of reducing — modern code often switches to warp shuffles for the last few passes, but the stock pattern above is plenty fast for most cases.

4. Cross-CTA aggregation via atomicAdd

Each CTA finishes its tree reduction with one scalar. We need to combine the scalars from all CTAs. Two real options:

  • Two-pass. Write per-block results to an array, then launch a small reduce-of-the-results kernel.
  • Atomic. Each block does atomicAdd(&global, blockResult). Used here. Fewer kernel launches, simpler code, slightly slower at very high CTA counts due to atomic contention.

atomicAdd serializes through the L2 cache: even though every CTA tries to update exp_sum[0] simultaneously, the hardware queues them. For softmax sized N=10,000 we have ~10 CTAs — atomic contention is negligible. For very large arrays you'd switch to the two-pass version.

5. The numerical-stability max trick

Softmax is invariant to additive shifts: subtracting any constant k from every input gives the same output (the constant cancels in the ratio). Picking k = max(x) makes every exp() argument ≤ 0, so every exp output is in (0, 1] — no overflow possible.

Note: x is read three times. That looks wasteful, but for a 10 KB array it sits entirely in L2 (1.5 MB) after the first pass, so passes 2 and 3 are L2 hits, not DRAM hits. The kernel is still dominated by the first DRAM read.

7. Subtle bits

  • 2 elements per thread. Each kernel issues two loads per thread (lines 49–55) and reduces both into one partial. This doubles the useful work per thread and halves the CTA count, saving launch overhead and atomic-contention.
  • __shared__ float data[256] — sized to blockDim.x exactly. Each thread owns one slot. No tile padding needed because we never do a "column" access pattern.
  • cudaMemset(d_sum, 0, ...) before launch — atomicAdd accumulates into whatever is there. Must zero between trials (lines 133–134).
  • One scalar broadcast across all threads — max_val[0] on line 47 is read by every thread of every CTA. It's a single 4-byte value sitting in L2; the broadcast is free.

8. Pitfalls

  • Forgetting to subtract max → exp(big number) → +inf → nan. The CPU reference (softmax.cu:11) does the same trick for the same reason.
  • Forgetting to cudaMemset the sum scalar → previous trial's value leaks in.
  • Misaligned tree reduction: if blockDim.x isn't a power of two, the range /= 2 loop drops elements. The kernel here uses 256 = 2⁸, so it's clean.
  • Atomic contention at very large N — the single global scalar becomes a serial bottleneck. Switch to a two-stage reduce.
