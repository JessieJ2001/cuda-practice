Sparse Matrix × Dense Matrix (SpMM)

TL;DR. C = A·B with A (M×N) ~65% zeros but stored dense, B (N×K) dense. The winning move is not to convert A to a sparse format — the conversion costs a full O(M·N) pass plus a prefix sum, which eats the savings when A is used once. Instead: keep A dense and skip the multiply when the element is zero. The trick that makes this fast is the thread layout — let a warp span the columns of C, and the zero-test becomes warp-uniform: every lane reads the same A element, takes the same branch, and skips together. Zero divergence, zero extra memory, no conversion.

Source: /workspaces/cuda-practice/spmm.cu. All line numbers below cite that file. Entry point: solve(const float* A, const float* B, float* C, int M, int N, int K, int nnz) — device pointers, result written to C.

1. The problem, and the trap

A is M×N and 60–70% of its entries are zero, but it arrives in plain row-major dense layout (plus an nnz count). B is N×K dense. We want C = A·B, M×K.

The textbook answer to "matrix is sparse" is "use a sparse format" (CSR) so you only ever touch the non-zeros. That's strategy 3 below — and it's a trap for a one-shot SpMM. Converting a dense A into CSR on the GPU requires:

  • a full O(M·N) read of A to count non-zeros per row,
  • a prefix sum over the M row-counts to get the CSR row pointers,
  • a second O(M·N) pass to scatter (column, value) pairs.

If you only multiply once, you've read A twice plus paid for a scan, just to avoid reading it once. The conversion is only worth it if the CSR form is reused many times. So this page builds three kernels and measures the trade-off:

2. Launch configuration

One thread per output element C[row][col]. The block is 16×16; x indexes the K columns of C, y indexes the M rows (spmm.cu:155–159): 

void solve(const float* A, const float* B, float* C, int M, int N, int K, int nnz) {    (void)nnz;  // not needed by the zero-skip strategy    dim3 block(TILE, TILE);                 // 16×16 = 256 threads    dim3 grid(cdiv(K, TILE), cdiv(M, TILE));  // x = K cols, y = M rows    spmmZeroSkip<<<grid, block>>>(A, B, C, M, N, K);}

Why x→columns and not x→rows? Because the warp is 32 threads with consecutive threadIdx.x. Mapping x to the columns of C is what makes both the coalescing and the divergence-free zero-skip work. That single decision is the whole kernel. §4 shows why.

3. Strategy 1 — the dense baseline

Plain GEMM, sparsity ignored, so we can measure what skipping actually buys (spmm.cu:46–57):

__global__ void spmmDenseNaive(const float* A, const float* B, float* C,                               int M, int N, int K) {    int col = blockIdx.x * blockDim.x + threadIdx.x;   // over K    int row = blockIdx.y * blockDim.y + threadIdx.y;   // over M    if (row >= M || col >= K) return;    float acc = 0.0f;    for (int n = 0; n < N; n++)        acc += A[row * N + n] * B[n * K + col];        // multiplies even when A==0    C[row * K + col] = acc;}

Correct, but it does the full M·N·K FMAs and — worse on this memory-bound problem — reads every B row for every output row, even where A is zero. ~65% of that B traffic is wasted.

4. Strategy 2 — zero-skip, and why it's divergence-free

One line longer than the baseline (spmm.cu:74–91):

__global__ void spmmZeroSkip(const float* A, const float* B, float* C,                             int M, int N, int K) {    int col = blockIdx.x * blockDim.x + threadIdx.x;   // over K    int row = blockIdx.y * blockDim.y + threadIdx.y;   // over M    if (row >= M || col >= K) return;    float acc = 0.0f;    const float* Arow = A + (size_t)row * N;    for (int n = 0; n < N; n++) {        float a = Arow[n];                 // warp-uniform broadcast load        if (a != 0.0f)                     // warp-uniform branch (no divergence)            acc += a * B[(size_t)n * K + col];   // coalesced over col    }    C[(size_t)row * K + col] = acc;}

Take one warp: 32 lanes with consecutive threadIdx.x and the same threadIdx.y. So every lane in the warp shares the same row, and they cover 32 consecutive col values. Now look at each line of the loop:

  • a = Arow[n] — row and n are identical across the warp, so all 32 lanes read the same address. The hardware serves this as a single broadcast load, cached in L1/L2.
  • if (a != 0.0f) — the condition depends only on (row, n), both warp-uniform. So the predicate is the same for all 32 lanes: the entire warp either does the work or skips it together. No warp divergence. This is the crux — a zero-skip that would normally shred a warp into divergent paths is here perfectly uniform, purely because the warp spans columns, not the sparse dimension.
  • B[n*K + col] — n is uniform, col is consecutive → consecutive addresses → one coalesced 128-byte transaction.
A warp spans columns of C, so the sparse dimension n is warp-uniform.

The standard objection to zero-skipping on a GPU — "data-dependent branches cause warp divergence" — does not apply here. The branch variable is the sparse element A[row][n], and every lane in the warp shares the same row and n. The branch is uniform by construction. With 65% zeros the warp skips ~65% of its coalesced B reads and FMAs, with no penalty.

Note nnz is unused by this kernel — it doesn't need to know how sparse A is, only which elements happen to be zero. We keep the parameter because the solve signature is fixed.

5. Strategy 3 — CSR conversion, and what it costs

The "proper" sparse approach. Build CSR from dense A in three kernels, then a CSR-SpMM that only touches non-zeros (spmm.cu:101–147): 

__global__ void csrRowCountKernel(...)  // thread/row: count nonzeros__global__ void exclusiveScanKernel(...) // rowPtr = prefix sum of counts__global__ void csrFillKernel(...)      // thread/row: scatter (col,val)__global__ void spmmCSRKernel(...)      // C[r][k] = Σ vals[i]·B[colIdx[i]][k]

The multiply (spmmCSRKernel) is genuinely less work — it loops only over a row's non-zeros, and the inner access B[colIdx[i]*K + col] is still coalesced over col. But getting there costs the conversion pipeline, and one stage is a real bottleneck:

The scan is the tax. Turning per-row non-zero counts into CSR row pointers is a prefix sum. The version in spmm.cu:111 is a deliberately simple single-thread serial scan — correct, but O(M) on one lane while the rest of the GPU idles. A production implementation would use the parallel Blelloch scan from topk/scan. Either way, you've now read A twice and run a scan to avoid reading it once. run-spmm-csr prints the build time and the multiply time separately so you can see the conversion dominate at these sizes.

The whole point: ~65% of the iterations skip the only two expensive things (the B DRAM read and the FMA) for the entire warp at once, while the cheap broadcast of A is what feeds the decision.

7. Why not shared-memory tiling?

A dense GEMM would tile B into shared memory to reuse it across rows. Here that helps less: the access into B is data-dependent (which n rows you touch depends on where A's non-zeros are), so a fixed tiling schedule loads B rows the warp then skips. The zero-skip kernel instead leans on the L2 cache: because every row's B[n*K+col] for a popular column n is the same line, the 1.5 MB L2 on the 1660 Super absorbs a lot of the reuse for free. Tiling is the next optimization if the problem were compute-bound; at 65% sparsity it is firmly memory-bound and skipping wins first.

8. Pitfalls and gotchas

  • Mapping x to rows instead of columns. Then a warp spans the M dimension, A[row][n] is different per lane, and the zero-test does diverge — plus the B read stops coalescing. The thread layout is the optimization.
  • Comparing floats with == 0.0f on generated data. Fine here because the sparsity pattern writes exact 0.0f. If zeros came from arithmetic you'd want a tolerance, not equality.
  • Sizing the CSR arrays. colIdx/vals must hold exactly nnz entries — that's why the problem gives you nnz up front (the CSR path needs it; zero-skip does not).
  • Serial scan at large M. The single-thread exclusiveScanKernel is correct but will dominate CSR build time as M grows; swap in a parallel scan for real use.
  • Integer overflow on indices. Row/col offsets are cast to size_t before multiplying by N/K so large matrices don't wrap a 32-bit index.
