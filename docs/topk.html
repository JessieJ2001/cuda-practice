Top-K Selection

TL;DR. Don't sort the whole array. Treat each float as a bit pattern and bin it by its top few bits in a shared-memory histogram. A tiny CPU loop walks the histogram to find which bucket holds the K-th element. Then drill into that bucket's bits with another pass. After ~8 passes the K-th value is pinned down; one final filter kernel sweeps the array and atomically grabs every element ≥ threshold.

Source: /workspaces/leetgpu-master/topk.cu.

DRAM L2 L1 / shared register warp

1. The problem (and why sorting is wrong)

Given N floats and a target K, return the K largest. The CPU does this with std::partial_sort in O(N log K) — fast, but inherently sequential.

On a GPU we have ~22 SMs sitting idle if we serialize. The real question becomes: can we find the K-th largest value without sorting? Yes — by repeatedly bucketing on bits of the input. The algorithm is called radix select.

2. The high-level algorithm

  1. Pass 1. Look at the top 4 bits of each float's bit pattern. Drop each element into one of 16 buckets. Count bucket sizes.
  2. CPU walks the histogram. Starting from the largest bucket (highest bits = biggest value), sum sizes until the running total reaches K. That bucket contains the K-th largest. Remember its prefix.
  3. Pass 2. Look at the next 4 bits, but only for elements whose top bits match the locked prefix. Build a 16-bucket histogram again.
  4. Repeat until only one element matches → that's the threshold value.
  5. Filter pass. Sweep the array; every element ≥ threshold joins the output via atomicAdd.

3. Inside radixSelectk — the histogram kernel

This is the workhorse, run once per radix pass (topk.cu:73–104): 

__global__ void radixSelectk(const float* input, int* output, int N, int prefix_val, int radix_offset) {    __shared__ int hist[NumBuckets];                       // per-CTA histogram (~16 ints)    if (threadIdx.x < NumBuckets) hist[threadIdx.x] = 0;    __syncthreads();    int shift = 32 - RadixBits - radix_offset;    uint32_t prefix_mask = ((1 << radix_offset) - 1) << (32 - radix_offset);    for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < N; i += gridDim.x * blockDim.x) {        float val = input[i];        uint32_t uval = reinterpret_cast<uint32_t&>(val);          // bit-cast float → uint32        if ((uval & prefix_mask) != prefix_val) continue;            // not in our search slice        auto bucket = (uval >> shift) & RadixMask;                  // extract radix digit        atomicAdd(&hist[bucket], 1);                              // shared-memory atomic    }    __syncthreads();    for (int histid = threadIdx.x; histid < NumBuckets; histid += blockDim.x) {        atomicAdd(&output[histid], hist[histid]);                  // merge into global hist    }}

Three patterns to call out:

Grid-stride loop

Line 85 doesn't do "one element per thread". Instead each thread starts at its global index and strides by gridDim.x * blockDim.x (the total number of threads) until it falls off the array. This lets one fixed grid (num_sms * 4 CTAs, line 149) chew through an array of any size — no per-call resize.

Per-CTA shared histogram

Every CTA gets its own hist[16] in shared memory. Each element does one atomicAdd against this private histogram. Shared-memory atomics are way faster than global atomics because they don't serialize across the chip — they only serialize across one SM.

One global merge at the end

After processing, threads 0..15 of each CTA atomically add their private histogram into the global one. 16 global atomics per CTA, not N atomics. This is the key optimization that makes histogramming on a GPU competitive.

4. The CPU loop that drives it (topk.cu:166–206)

The host walks the histogram each pass to decide which bucket to drill into. This is fundamentally serial — but it's 16 bucket comparisons, not N comparisons. The kernel does the parallel work, the CPU does the tiny control flow. 

cur_k = N + 1 - K;        // we're looking for the K-th-largest, i.e. (N-K+1)-th smallest
prefix_val = 0;
for radix_offset in (0, 4, 8, …, 28):
    launch zero(hist)
    launch radixSelectk(input, hist, N, prefix_val, radix_offset)
    cudaMemcpy hist → host
    sum = 0
    for b in 0..15:
        if sum + hist[b] ≥ cur_k:           // this bucket contains the K-th element
            prefix_val |= b shifted into position
            cur_k -= sum
            if hist[b] == 1:                // unique → we have the value
                launch locateValue(...)
            break
        sum += hist[b]

Note the cudaMemcpy in the middle of the loop — that's a device-to-host round-trip every pass. It's the cost of doing the bucket-selection decision on the CPU. Up to 8 passes × ~5 μs each, so ~40 μs of overhead. For huge N (millions+) the kernel time dominates and this is fine. For small N you'd want to do the decision on the GPU too.

5. The final filter pass (topk.cu:122–131)

Once the threshold value is pinned down, one more grid-stride kernel sweeps the array. Every element ≥ threshold competes for a slot in the output via a single global atomic counter: 

__global__ void filterK(const float *input, ValueIndex *output, int *p_out_idx, int N, float selectk) {    for (int i = blockIdx.x * blockDim.x + threadIdx.x; i < N; i += gridDim.x * blockDim.x) {        float val = input[i];        if (val >= selectk) {            auto out_idx = atomicAdd(p_out_idx, 1);            output[out_idx] = ValueIndex{val, i};        }    }}

The output is unsorted — order depends on which CTA's thread happened to win each atomic. That's fine for top-K (the order within the top K usually doesn't matter; compare with the CPU's topkCPU which sorts only because it can).

The first radixSelectk pass is the only one that pays the full DRAM bandwidth cost. All later passes find input bytes still in L2. So this algorithm runs at ~336 GB/s × 1 regardless of K — the K parameter affects only the tiny CPU loop.

7. Subtle bits

  • Bit-cast trick. For positive IEEE-754 floats, the bit pattern is monotonic — larger float = larger uint32. This is what lets us do radix sort on float bits without floating-point comparison instructions. (Negative floats break monotonicity; a real production version XORs sign-bit-related fixups.)
  • Why num_sms * 4 CTAs. (Line 149.) Enough to fill every SM with multiple resident blocks for latency hiding. Going wider just costs more atomic contention without speeding up the grid-stride loop.
  • Shared-memory atomics on Turing are fast (~1 cycle when uncontended) but Bank-conflicts and same-address contention cause serialization. The comment at line 95 calls this out — for skewed input distributions one bucket gets hammered.
  • Atomic on a global counter for output indices is the same pattern as a single-producer / multi-consumer queue. It works because K is small; if K were N/2 the atomic would become a bottleneck.

8. Pitfalls

  • Forgetting to zero the histogram between passes — the kernel zero at topk.cu:65 is called every iteration.
  • Negative floats — the current code assumes positives, which is what the test harness generates.
  • K close to N — atomic on the output counter becomes a bottleneck.
  • Many duplicates of the threshold value — the filter could emit more than K elements. Production code adds a "stop at K" check; the source here doesn't.
