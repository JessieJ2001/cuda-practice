Matrix Transpose

TL;DR. The naive transpose reads coalesced rows but writes scattered columns, costing ~32× the DRAM bandwidth it should. The fix: stage a 32×32 tile in shared memory so both the read and the write are coalesced. One subtlety — declare the tile as tile[32][33], not tile[32][32], to dodge a 32-way bank conflict.

Source: /workspaces/leetgpu-master/transpose.cu. All line numbers below cite that file.

DRAM L2 L1 / shared register warp

1. The problem

Given an R×C matrix A, produce its transpose AT of size C×R: element (i, j) of the input becomes element (j, i) of the output. In row-major memory: 

output[col * rows + row] = input[row * cols + col];

No arithmetic at all — every byte read becomes a byte written. This is the cleanest possible memory-bound kernel: there is nothing to optimize except memory bandwidth.

Bandwidth ceiling. For a 1024×1024 float matrix we move 2 × 1024 × 1024 × 4 B = 8 MB of data. At 336 GB/s we can do that in ≈ 24 μs. Any slower means we're wasting bandwidth — the only thing that matters.

2. Launch configuration

Both kernels use the same launch (transpose.cu:60–63): 

dim3 blockSize(TILE_SIZE, TILE_SIZE);             // 32 × 32 = 1024 threads / CTA// rows = 96, cols = 256// gridSize = (8, 3)dim3 gridSize(cdiv(cols, TILE_SIZE), cdiv(rows, TILE_SIZE));

For a 1024×1024 input that's a 32 × 32 grid of 32 × 32 blocks — 1024 CTAs, 1024 threads each, 1,048,576 threads total. The 1660 Super has 22 SMs, so multiple CTAs co-reside per SM and the GPU keeps thousands of warps in flight.

Each CTA processes one 32×32 tile of the output. (Schematic — diagram shows 8×8 grid and 8×8 block for legibility.)

3. The naive kernel — why it's slow

The implementation is two lines of memory work (transpose.cu:22–30):

__global__ void transposeNaive(const float* input, float* output, int rows, int cols) {    // Simple approach - may have coalescing issues    int row = blockIdx.y * blockDim.y + threadIdx.y;    int col = blockIdx.x * blockDim.x + threadIdx.x;    if (row < rows && col < cols) {        output[col * rows + row] = input[row * cols + col];   // the whole kernel    }}

A warp here is one row of the 32×32 block: 32 threads with the same threadIdx.y and consecutive threadIdx.x. Look at what they touch:

  • Read: thread i reads input[row*cols + coli]. Consecutive threads → consecutive columns → consecutive addresses. Coalesced. One DRAM transaction.
  • Write: thread i writes output[coli*rows + row]. Consecutive coli means addresses jump by rows bytes (= 4096 B for 1024×1024). Scattered. Up to 32 separate transactions.
One warp does one fat read and a fan of scattered writes.
step 1 / 5
For a 1024×1024 transpose, the naive kernel moves roughly 32× more bytes than it had to on the write side. Measured bandwidth lands far below the 336 GB/s ceiling — typically 30–60 GB/s on this card.

4. The shared-memory kernel

Trick: don't write to DRAM in column order. Instead, park a whole 32×32 tile in shared memory first, then write it back to DRAM in row order. Both DRAM trips are now coalesced — DRAM never sees a strided access. 

__global__ void transposeShared(const float* input, float* output, int rows, int cols) {    __shared__ float tile[(TILE_SIZE + 1) * TILE_SIZE];   // ← +1 = bank-conflict trick    int row = blockIdx.y * blockDim.y + threadIdx.y;    int col = blockIdx.x * blockDim.x + threadIdx.x;    if (row < rows && col < cols)        tile[threadIdx.y * (TILE_SIZE + 1) + threadIdx.x] = input[row * cols + col];   // coalesced read    __syncthreads();   // wait for every thread in the CTA    int outrow = blockIdx.x * blockDim.x + threadIdx.y;    int outcol = blockIdx.y * blockDim.y + threadIdx.x;    if (outrow < cols && outcol < rows) {        float out = tile[threadIdx.x * (TILE_SIZE + 1) + threadIdx.y];   // transposed read from shared        output[outrow * rows + outcol] = out;   // coalesced write    }}

What happened in each step:

  1. Coalesced load from DRAM into the shared tile. Thread (tx, ty) stores at tile[ty][tx]. Within a warp ty is fixed and tx spans 0..31 — consecutive shared addresses, no conflict.
  2. __syncthreads() — every thread in the CTA waits here. After this point, the entire 32×32 tile is sitting in SRAM.
  3. Transposed read from shared. Now thread (tx, ty) reads tile[tx][ty]. Within a warp ty is fixed and tx spans 0..31 — that's a column of the shared tile. Bank conflict alert! → see §5.
  4. Coalesced store back to DRAM. outrow is fixed per warp, outcol spans 0..31 — consecutive DRAM addresses.
Both DRAM trips are coalesced. The "weird" access becomes a fast SRAM op instead.
step 1 / 6

5. Bank conflicts and the +1 trick

Shared memory is split into 32 banks, each 4 bytes wide. Word at offset k lives in bank k % 32.

If we declared the tile as a plain tile[32][32], then a column read — thread i reads tile[i][fixed_col] — touches addresses i*32 + fixed_col. Those all share the same bank index. A 32-way conflict. The hardware serializes all 32 reads.

By declaring the tile (TILE_SIZE + 1) wide instead, the row stride becomes 33 (= 32 + 1). Now bank index for a column read is (i*33 + c) % 32 = (i + c) % 32 — each thread hits a different bank. Conflict-free, single-cycle.

Step through the two layouts. Same kernel, one character difference, 8× slowdown.
step 1 / 3
Memorize this pattern. Any time you transpose data through shared memory, the tile gets one extra column. Costs a few hundred bytes, earns you a full 32× shared-memory throughput.

Compare to the naive kernel which skips the middle three rows entirely and pays the DRAM cost twice instead, with the second trip uncoalesced.

7. Why the shared-memory version is fast

  • DRAM is touched in row order both ways. Every warp's DRAM load and store coalesces into one 128-byte transaction.
  • The transposed access pattern moves to shared memory — which can serve any access pattern in 1 cycle, as long as banks don't collide.
  • The +1 padding dodges the only remaining bank-conflict landmine.
  • Latency hiding: 1024 threads/CTA × many CTAs/SM keeps dozens of warps in flight per SM. When one warp is waiting on DRAM, another is ready.

8. Pitfalls and gotchas

  • Forgetting __syncthreads() → reads from the tile before all writes have landed. Random garbage output.
  • Putting __syncthreads() inside an if that's not uniform across the CTA → undefined behavior, possible deadlock.
  • Rectangular tiles (e.g. 32×16): the bank-conflict math changes — the safe +1 trick still works because gcd(stride, 32) shifts banks.
  • Non-multiple sizes: the boundary ifs in the kernel handle ragged edges, but you still launch full warps for the partial blocks — some lanes are idle.
