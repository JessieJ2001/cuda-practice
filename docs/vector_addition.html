Vector Addition

TL;DR. C[i] = A[i] + B[i] for every i. The simplest possible CUDA kernel. We use it to nail down the warp execution model and what perfect coalescing looks like — every kernel after this is a deviation from this baseline.

Source: /workspaces/leetgpu-master/vector_addition.cu.

DRAM L2 L1 / shared register warp

1. The problem

Given two vectors A and B of length N, compute C[i] = A[i] + B[i]. The CPU version is one for-loop; the GPU version is the same loop but every iteration runs in parallel. 

// CPU                              // GPU
for (int i = 0; i < N; i++)         every thread does
    C[i] = A[i] + B[i];                C[idx] = A[idx] + B[idx];
                                       (with idx = its global thread index)

2. Launch configuration

1D grid, 1D block. From vector_addition.cu:38–39:

int threadsPerBlock = 1024;int blocksPerGrid = (N + threadsPerBlock - 1) / threadsPerBlock;   // ceil(N/1024)

For N = 1,000,000 that's 977 blocks × 1024 threads = 1,000,448 threads. The last 448 threads run with idx ≥ N and are guarded out by the bounds check on line 22. (Launching extra threads is cheaper than launching a second kernel for the tail.)

3. The kernel itself

__global__ void vectorAddGPU(const float* A, const float* B, float* C, int N) {    // global thread index — unique across the entire grid    int idx = blockIdx.x * blockDim.x + threadIdx.x;    if (idx < N) {        C[idx] = A[idx] + B[idx];    }}

That's the whole kernel. Three loads, one add, one store. Every thread does exactly the same work on a different index. No shared memory, no __syncthreads, no shuffles — nothing to share.

5. Memory-hierarchy story

Every warp moves 384 bytes of DRAM traffic (128 in for A, 128 in for B, 128 out for C) and does 32 floating-point adds. The ratio of compute to memory traffic is comically low — 0.083 FLOP / byte. This kernel is the canonical memory-bound example.

6. Bandwidth math (the only number that matters)

For N = 1,000,000 floats:

  • Bytes moved: 3 × N × 4 = 12 MB
  • 1660 Super DRAM ceiling: 336 GB/s
  • Lower bound on time: 12 MB ÷ 336 GB/s ≈ 36 μs

On this card the kernel typically hits ~28–32 μs — within shouting distance of the ceiling. There's nothing to optimize except making sure you didn't accidentally break coalescing.

The mental ratio to remember: 1 FP add ≈ 12 bytes of DRAM traffic for this kernel. As long as compute is cheap and memory is expensive (which is the universal case on this card), kernels like vector add will live in the memory-bound regime, and the GPU's 5 TFLOPS of FP32 is mostly idle.

7. Why this kernel is "boring" (and that's the lesson)

  • No shared memory. Nothing to share — each thread's element is independent.
  • No bank conflicts — no shared memory.
  • No divergence. The only if is the bounds check at the very end of the grid; threads in a warp either all pass or (for the last warp) divide cleanly between in-bounds and out-of-bounds.
  • Perfect coalescing. Thread i in a warp reads address base + 4i for every load. The hardware merges into one 128-byte transaction per array per warp.

Everything that follows on this site is a story about losing one of these properties and earning it back. Transpose loses coalescing on the write and earns it back with shared memory. Softmax loses parallelism for the reduction and earns it back with a tree. Flash attention loses re-use and earns it back with online softmax. Vector add is the baseline: this is what the GPU does when nothing gets in its way.
